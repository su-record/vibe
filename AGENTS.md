## Hard Rules

### Behavior
- **Modify only requested scope** ‚Äî Every changed line traces to the user's request
- **Edit existing files over creating new** ‚Äî Fix at source
- **Preserve existing style** ‚Äî Match conventions even if you'd do it differently
- **Respect Ctrl+C / Escape** ‚Äî Previous task CANCELLED
- **State assumptions, ask when uncertain** ‚Äî Don't pick silently when ambiguity exists; push back if a simpler approach exists

### Goal-Driven Execution
Transform imperative tasks into verifiable goals **before** coding:

| Instead of | Transform to |
|---|---|
| "Add validation" | "Tests for invalid inputs pass" |
| "Fix the bug" | "A test reproducing it passes" |
| "Refactor X" | "Tests pass before and after" |

Weak criteria ("make it work") require constant clarification. Strong criteria let the loop run independently.

### TypeScript (enforced by Quality Gate hooks)
- No `any` / `as any` / `@ts-ignore` ‚Äî use `unknown` + type guards; fix at root
- Explicit return types on all functions

### Complexity Limits
Function ‚â§50 lines ¬∑ Nesting ‚â§3 ¬∑ Params ‚â§5 ¬∑ Cyclomatic ‚â§10

### Forbidden Patterns
No `console.log` in commits ¬∑ No hardcoded strings/numbers ¬∑ No commented-out code ¬∑ No incomplete code without TODO

### Convergence (review / auto-fix loops)
- **Loop until P1 = 0 AND no new findings** ‚Äî no round cap
- **Narrowing scope**: Round 1 full ‚Üí Round 2 P1+P2 ‚Üí Round 3+ P1 only
- **Stuck detection** (same findings/score 2 rounds in a row) ‚Üí ask user (fill values / approve sub-100 / abort). Never silently proceed sub-100
- **`ultrawork` exception** ‚Äî skip user prompt; record gaps as TODO to stay non-interactive
- **Changed files only** ‚Äî never full-project scan

## Context Portability

Vibe's thesis: **the harness can be closed ‚Äî the context layer must not be**. Every piece of context is a plain file you own, version in git, and carry to another harness. Claude Code happens to be one runtime; coco is another; future harnesses follow the same contract.

### Portable artifacts (all plain markdown/JSON, git-tracked)
| Artifact | Path | Consumed by |
|---|---|---|
| Project instructions | `CLAUDE.md` (SSOT) ¬∑ `AGENTS.md` (mirror) | Claude Code ¬∑ coco |
| Skills | `~/.claude/skills/*/SKILL.md` ¬∑ `~/.coco/skills/` | Both (same SKILL.md format) |
| Commands | `~/.claude/commands/*.md` ¬∑ `~/.coco/commands/` | Both |
| Memory | `~/.claude/projects/<slug>/memory/*.md` + `MEMORY.md` index | Claude Code today; portable by path remap |
| Project data | `.claude/vibe/{plans,specs,features,research,todos}/` | Any harness via file read |

### Portability contract
1. **CLAUDE.md is the content SSOT.** `AGENTS.md` is regenerated by `/vibe.docs agent` ‚Äî never edit it directly.
2. **Skills and commands are harness-neutral markdown.** The `.claude/` vs `.coco/` split is installation path only; file formats are identical.
3. **Memory is a flat directory of markdown.** No proprietary DB, no vendor API. A new harness adopts memory by pointing at the directory.
4. **Project state lives under `.claude/vibe/`** (not `~/.claude/`). Cloning the repo gives a new machine everything it needs.

### What is NOT portable (accepted lock-in)
- `.claude/settings.local.json` hooks ‚Äî Claude Code-specific shell integration
- `better-sqlite3` agent daemon DB ‚Äî runtime state, not context
- Claude Code's built-in slash commands (`/clear`, `/new`) ‚Äî harness primitives

Rule of thumb: if losing the harness tomorrow would lose the artifact, it is not context ‚Äî it is runtime. Keep runtime in `.claude/settings.local.json` and databases; keep everything else in markdown.

## Architecture (Non-Obvious)

### Module System
- ESM only (`"type": "module"`) ‚Äî imports need `.js` extension
- Build before test: `npm run build && npx vitest run`

### Config Locations
| Path | Purpose |
|---|---|
| `~/.vibe/config.json` | Global credentials, models (0o600) |
| `.codex/vibe/config.json` | Project stacks, capabilities |
| `.codex/settings.local.json` | Project hooks (auto-generated, don't commit) |

### Gotchas
- `better-sqlite3` WAL mode ‚Äî synchronous API
- `crypto.timingSafeEqual` requires same-length buffers ‚Äî check length first
- **Stack ‚Üí asset SSOT**: `GLOBAL_SKILLS_*`, `STACK_TO_SKILLS`, `CAPABILITY_SKILLS` in `src/cli/postinstall/constants.ts`
- **Hook dispatch order**: `prompt-dispatcher.js` ‚Üí `keyword-detector.js` ‚Üí `llm-orchestrate.js`

## Workflow

`/vibe.spec` is the single entry point ‚Äî orchestrates interview ‚Üí plan ‚Üí spec ‚Üí review ‚Üí `/vibe.run` ‚Üí `/vibe.verify` ‚Üí `/vibe.trace`. For UI types (website/webapp/mobile), `/vibe.figma` branches in parallel. Smart Resume detects existing `.codex/vibe/{interviews,plans,specs}/*.md` to skip phases.

| Task Size | Approach |
|---|---|
| 1‚Äì2 files | Plan Mode |
| 3+ files | `/vibe.spec` |

## Magic Keywords

| Keyword | Effect |
|---|---|
| `ultrawork` / `ulw` | Parallel agents + auto-continue + Ralph Loop |
| `ralph` | Iterate to 100% (no scope reduction) |

## Skill Tiers

3 tiers prevent context overload: **core** (always active) / **standard** (project-setup selected) / **optional** (explicit `/skill` only). SSOT: `GLOBAL_SKILLS_CORE` + `GLOBAL_SKILLS_STANDARD` in `src/cli/postinstall/constants.ts`. Proactive triggers live in each skill's frontmatter.
