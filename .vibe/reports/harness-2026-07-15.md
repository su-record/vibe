## 모델 업그레이드 관점의 발견

### 1. 올바른 폐기 판단 장치는 있지만 휴면 상태다

`ParityTester`는 이전 모델 baseline, 새 모델 baseline, with-skill 성능을 비교하도록 설계되어 있다. pass rate, duration, token도 기록한다. 스킬 삭제·축소는 이 근거를 따라야 한다.

현재 간극:

- model parity를 실행하는 production command나 lifecycle 단계가 없다.
- 로컬 model baseline/eval record가 없다.
- 실제 폐기 판정에는 0.85 parity ratio만 반영된다.
- 최소 eval 개수와 최소 개선 폭 상수가 적용되지 않는다.
- 그대로 연결하면 불충분한 데이터로 폐기 권고를 만들 수 있다.

### 2. 정적 모델 티어가 기술 부채가 되었다

- 에이전트 파일 62개와 스킬 파일 11개가 Haiku/Sonnet/Opus를 참조한다.
- `ModelRouter`는 키워드 점수를 세 고정 티어로 매핑한다.
- 에이전트 상수에는 Claude 4.5, GPT 5.3 Codex, 그보다 최신 모델 문자열이 혼재한다.
- 루트 지침도 탐색 → Haiku, 구현 → Sonnet, 아키텍처 → Opus로 고정한다.

워크플로 계약의 모델명은 `fast-search`, `code-change`, `deep-reasoning`, `visual` 같은 측정 가능한 capability profile로 바꾸는 편이 낫다. provider/model 선택은 프롬프트 정체성이 아니라 설정과 benchmark 데이터가 담당해야 한다.

### 3. 에이전트 수가 한계 효용보다 높은 점수를 받고 있다

- `vibe.harness` 자체가 결정론적 파일시스템 인벤토리에 explorer agent 3개를 의무화한다.
- `vibe.review`는 12개 이상 reviewer를 병렬 실행하고 UI 변경이면 더 추가한다.
- explorer, implementer, architect마다 low/medium/high 프롬프트 변형이 있다.
- 스킬·에이전트 전체 텍스트 표면은 약 779KB다.

강한 frontier model 기준에서 결정론적 인벤토리는 스크립트가 맡고, review는 위험 기반으로 선택해야 한다. 독립 reviewer는 서로 다른 위협 모델을 볼 때 유효하며, 같은 작업을 역할명만 바꿔 반복할 때는 가치가 낮다.

### 4. 지시 충돌은 지시 부족보다 더 위험하다

함수 길이 20줄 대 50줄 충돌이 구체적인 예다. 강한 모델은 어느 쪽이든 더 잘 따를 수 있지만, 충돌한 소스 중 무엇이 권위 있는지는 알 수 없다. 루트 문서와 대형 스킬에 loop, resume, approval, score 절차가 반복되는 것도 같은 위험이다.

불변 조건은 하나의 machine-readable SSOT로 옮기고, 스킬은 입력·출력·예외·포인터 중심으로 줄여야 한다.

### 5. 결정론적 경계는 제거가 아니라 복구 대상이다

활성 로컬 프로젝트에는 설정된 훅과 project `.vibe/config.json`이 없고, permission allow-list는 97개로 넓다. 모델 향상은 scope, 위험 명령, test, publish 경계를 약화할 이유가 아니다. 현재는 활성화된 기계적 집행보다 soft instruction이 더 많은 상태로, 바람직한 방향과 반대다.

| 판단 | 대상 |

| 즉시 단순화 | 반복된 loop/resume 설명, 정적 모델명 지침, agent 3개가 수행하는 결정론적 인벤토리, 무조건적 BDD 생성, 300줄 초과 스킬의 반복 예시 |
| 제거 전 측정 | low/medium/high agent 변형, 12+ reviewer fan-out, memory/RAG/reflection 중복, evolution/recipe/regress 중복, domain skill |

## 우선 개선 3가지

1. **하나의 결정론적 quality gate를 green으로 복구한다**
   - build, test, skill declaration, count, catalog freshness, 복구된 RTM을 한 명령으로 실행한다.
   - 같은 명령을 로컬, `vibe.verify`, CI에서 사용한다.
   - 먼저 Windows/POSIX 테스트 가정과 함수 길이 20/50 충돌을 해결한다.

2. **가지치기 전에 model-upgrade parity를 운영 가능하게 만든다**
   - `SkillEvalRunner` → old/new baseline → with-skill run → `ParityTester` → `DeprecationDetector`를 명시적 command로 연결한다.
   - 충분한 eval 수, 반복 실행, pass-rate 비회귀, token/latency 개선을 모두 만족해야 retire하게 한다.
   - 한계 비용을 측정하기 쉬운 `vibe.harness`, agent tier 변형, core reviewer부터 시작한다.

3. **추론 보정을 위험 기반 routing으로 축소한다**
   - Haiku/Sonnet/Opus 고정 역할을 capability profile과 benchmark 기반 provider 선택으로 교체한다.
   - 12개 이상 항상 실행되는 review를 소수 기본 reviewer + 변경 위험에 따른 security/data/UI specialist로 바꾼다.
   - 기본은 AC를 포함한 단일 SPEC으로 두고, 실행 가능한 행동 계약이 필요할 때만 별도 Feature를 만든다.

Vibe에는 이미 충분한 장치가 있다. 다음 개선은 에이전트나 규칙 추가가 아니라 근거에 따른 제거여야 한다. 먼저 결정론적 게이트를 복구하고, 기존 parity/eval 인프라로 강한 모델에서 각 스킬의 한계 가치를 측정한 뒤, pass rate 개선이 token·latency 비용을 정당화하지 못하는 프롬프트와 fan-out만 폐기해야 한다.
