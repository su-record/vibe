## What This Project Does

Vibe is an npm-distributed AI coding quality harness. It is not primarily an application runtime. It installs and operates a controlled working environment around Claude Code, Codex, Cursor, and Antigravity so AI coding work can move through planning, implementation, verification, review, memory, and recovery loops.

The product has four main surfaces:

1. CLI surface: `vibe init`, `vibe update`, `vibe setup`, `vibe codex`, `vibe figma`, `vibe stats`, `vibe config`, provider auth commands, and install/update/remove flows.
2. Harness assets: `skills/`, `agents/`, `vibe/rules/`, `vibe/templates/`, `languages/`, and hook scripts shipped through the package.
3. Runtime hooks: prompt dispatch, pre-tool guard, post-edit quality detection, scope guard, verify ledger, auto-test, stop/notify, and Codex notify integration.
4. Tool API exports: memory, semantic analysis, convention checks, UI helpers, PRD/SPEC tools, loop validation, orchestration helpers, and telemetry.

The main user flow is:

1. `npm install -g @su-record/vibe`
2. `postinstall` copies global assets into CLI-specific locations and installs global rules.
3. `vibe init` detects the project stack, creates `.vibe/`, writes `CLAUDE.md` or `AGENTS.md`, installs project hooks, local skills, local agents, language rules, and config.
4. The user invokes `/vibe` in Claude Code or `$vibe`/`/skills` in Codex.
5. Skills route the task through SPEC, run, verify, trace, review, regress, Figma, design, harness, or analysis paths.
6. Hooks and ledgers enforce scope, dangerous command checks, quality findings, verify-required state, and completion notifications.

## Product Consistency

Score: 72/100

Important drift:

- `README.md` says 40+ agents and 44 skills.
- `package.json` description says 60+ agents and 69 skills.
- `validate:skill-invocation` reports 70 skills.

This is not only documentation polish. The project sells itself as a harness with a curated skill/agent surface, so count drift makes installation behavior and marketing claims harder to trust.

The repository also has `.vibe/` runtime state and active scope, but no `.vibe/config.json` at analysis time. The packaged `vibe/config.json` exists. If this repository itself is meant to be managed by Vibe, missing project `.vibe/config.json` weakens scope-guard and behavior-key clarity.

## Top Findings

1. P1: Dependency security gate is failing.
   - `npm audit` found 10 vulnerabilities, including 1 critical direct dev dependency (`vitest`) and 5 high transitive vulnerabilities.
   - Recommended next action: update dev-toolchain dependencies in one focused dependency remediation change, then run `npm audit`, typecheck, test, skill-doc check, and skill invocation validation.

2. P2: Core orchestration functions exceed the project's own complexity rules.
   - The most severe case is `src/cli/detect.ts` with complexity above 100 in stack detection.
   - Recommended next action: characterize current stack detection behavior with tests, then split detector data from detector control flow.

3. P2: Install/update dependency direction is too tangled.
   - `ProjectSetup.ts` depends on postinstall mappings while postinstall imports project setup generation helpers.
   - Recommended next action: extract shared asset maps and doc-generation helpers into dependency-neutral modules.

4. P2: Test suite passes but emits noisy expected-warning output.
   - UI/UX CSV loader tests repeatedly print missing-file warnings.
   - Recommended next action: make expected missing fixture checks silent or assert warnings explicitly.

5. P2: README/package/skill catalog counts disagree.
   - This creates trust drift for a product whose main promise is managed harness capability.
   - Recommended next action: generate these counts from the same source used by `validate:skill-invocation`.

## Suggested Follow-Up

1. Dependency remediation: fix `npm audit` first because it is the only failing deterministic gate found during analysis.
2. Complexity remediation: start with `src/cli/detect.ts`, because it is both central and the largest outlier.
3. Architecture cleanup: remove circular dependencies around project setup/postinstall and phase pipeline stores.
4. Documentation consistency: make README capability counts generated or checked.
5. Test output cleanup: reduce expected stderr noise in UI/UX CSV tests.
