# Claude Code 소스 분석 → vibe 적용 사항

분석일: 2026-04-09
대상: Claude Code 소스 코드 분석서 (2026-03-31)

## 핵심 요약

Claude Code의 설계 원칙 3가지: **안전성, 성능, 확장성**.
vibe에 적용 가능한 것을 **즉시 적용(P1)**, **단기 적용(P2)**, **참고만(P3)**으로 분류.

---

## P1: 즉시 적용 (현재 작업에 직접 영향)

### 1. 도구 동시성 파티셔닝 — figma-extract 이미지 다운로드

**Claude Code 패턴**: 안전한 도구(Read, Grep)는 최대 10개 병렬, 위험한 도구(Edit, Bash)는 순차 실행.

**vibe 현황**: `BackgroundManager.ts`에 모델별 동시성 제한 있으나, **도구 유형별 safe/unsafe 분류 없음**. 모든 도구를 동일하게 취급.

**적용 방안**: `figma-extract.js`의 이미지 다운로드에 적용.
- tree 추출 (읽기 전용) → 병렬 OK
- 이미지 다운로드 (읽기 전용) → 병렬 OK  
- screenshot 렌더링 (API rate limit) → 순차 + 500ms 간격
- 현재 `Promise.all`로 전부 병렬인데, screenshot은 rate limit에 걸림
- **개선**: download는 병렬, screenshot은 순차 큐로 분리

### 2. 에러 보류(Withholding) & 복구 — API 호출 탄력성

**Claude Code 패턴**: 복구 가능한 에러는 사용자에게 즉시 보여주지 않고, 자동 복구를 먼저 시도.
- 413 → 컨텍스트 축소 시도 → 전체 요약 시도 → 그래도 안 되면 에러 표면화
- 출력 토큰 초과 → 한도 에스컬레이션(8K→64K) → 최대 3번 이어쓰기

**vibe 현황**: `llm-orchestrate.js`에 retry + fallback 있으나, **에러 보류 패턴 없음**. 에러 즉시 사용자에게 전달.

**적용 방안**: `figma-extract.js`와 `llm-orchestrate.js`에 적용.
- Figma API 429 → 자동 대기 후 재시도 (현재 구현됨)
- Figma API 500 → 에러 보류, scale 다운(2→1) 시도 후 실패 시만 보고
- LLM 호출 → 컨텍스트 축소 후 재시도 전략 추가

### 3. 스트리밍 도구 실행기 — API 응답 + 도구 실행 겹치기

**Claude Code 패턴**: API 응답이 스트리밍 중일 때, 이미 도착한 도구 블록의 실행을 미리 시작.

**vibe 현황**: `stream.ts`에 스트리밍 파서 있으나, **오케스트레이터와 미통합**. 응답 완료 후 도구 실행.

**적용 방안**: vibe.figma Phase 2에서 tree 추출 완료 즉시 이미지 다운로드 시작 (현재는 tree 완료 → 이미지 시작이 순차적). 작은 변경으로 Phase 2 시간 단축 가능.

## P2: 단기 적용 (스킬/프레임워크 개선)

### 4. 자동 압축 서비스 — 긴 세션 안정성

**Claude Code 패턴**:
- 토큰 사용량 > (컨텍스트 윈도우 - 13,000) → 자동 압축
- 회로 차단기: 3회 연속 실패 → 압축 시도 중단
- 압축 후 최근 참조 파일 5개 복원 (50K 토큰 예산)
- 압축 후 스킬 재주입 (25K 예산, 스킬당 5K)

**vibe 현황**: `ContextCompressor.ts` 있으나 **수동 트리거**, 자동 발동 없음. 압축 후 스킬 재주입 없음.

**적용 방안**:
- vibe.figma 세션은 Phase 0~6까지 매우 길어 컨텍스트 한계 도달 빈번
- 자동 압축 트리거를 오케스트레이터에 통합
- 압축 시 **현재 Phase 번호 + 진행 상태** 를 보존 (스킬 재주입처럼)
- 회로 차단기 패턴 추가

### 5. 코디네이터 모드 — vibe 병렬 에이전트 고도화

**Claude Code 패턴**:
- 리더는 직접 코드 수정 안 함, 분배만
- 4단계: Research(병렬) → Synthesis(순차, 리더 필수) → Implement(영역별) → Verify(병렬)
- 핵심 규칙: "리더는 반드시 워커 결과를 직접 이해해야 함"

**vibe 현황**: `dispatching-parallel-agents` 스킬로 병렬 에이전트 사용. 그러나:
- Research/Synthesis 구분 없음
- 리더가 결과를 이해하지 않고 넘어갈 수 있음
- 영역별 충돌 방지 로직 없음

**적용 방안**: vibe.figma에 직접 적용.
- Phase 2 추출: 워커 병렬 (MO/PC 동시 추출)
- Phase 3 리매핑: 리더 순차 (MO↔PC 매칭은 전체 맥락 필요)
- Phase 4 코드 생성: 영역별 (섹션당 1 워커, 충돌 방지)
- Phase 6 검증: 워커 병렬 (독립 섹션별 검증)

### 6. 메모리 시스템 타입화 — 메모리 품질 향상

**Claude Code 패턴**: 메모리 4유형 명확 분리.
- user: 누구인지 (역할, 전문성, 선호)
- feedback: 어떻게 일하는지 (교정 + 확인, Why 포함)
- project: 무슨 일이 일어나고 있는지 (목표, 기한)
- reference: 어디를 봐야 하는지 (외부 시스템 포인터)
- **저장하면 안 되는 것**: 코드 패턴, 아키텍처, git 히스토리, 디버깅 레시피

**vibe 현황**: 메모리 시스템 있으나 타입 분류가 느슨. "무엇이든 저장" 경향.

**적용 방안**: CLAUDE.md의 메모리 가이드라인에 Claude Code의 타입 체계를 이미 반영함 (auto memory). 스킬 레벨에서도 **feedback 메모리 자동 감지** 추가 — 사용자가 "아니 그게 아니라"라고 하면 자동으로 feedback 메모리 생성.

### 7. 권한 파이프라인 — 훅 디스패치 강화

**Claude Code 패턴**: 10단계 파이프라인.
1. 도구 검색 → 2. 중단 확인 → 3. 입력 검증(Zod) → 4. PreToolUse 훅
→ 5. 권한 확인 → 6. 실행 → 7. 결과 매핑 → 8. 대형 결과 디스크 저장
→ 9. PostToolUse 훅 → 10. 텔레메트리

**vibe 현황**: 훅 디스패치 체인 있으나 **입력 검증 단계 없음**, 대형 결과 디스크 저장 없음.

**적용 방안**:
- Pre-tool 가드에 **Zod 스키마 검증** 추가 (현재는 패턴 매칭만)
- LLM 응답이 너무 클 때 디스크 저장 + 참조 전달 패턴 적용

## P3: 참고 (아키텍처 인사이트)

### 8. 시작 성능 최적화

**Claude Code 패턴**: main.tsx 800KB 단일 파일 (디스크 읽기 1회), 병렬 I/O 사전 실행 (-65ms), 조건부 모듈 로딩 + 빌드 시 dead code 제거.

**vibe 인사이트**: `vibe init`/`vibe update`의 시작 속도 개선에 참고. 현재 postinstall에서 파일 복사가 순차적인데, 병렬화 가능.

### 9. 이중 버퍼링 TUI

**Claude Code 패턴**: 게임 엔진식 이중 버퍼링 + 변경 셀만 출력 + 프레임 조절.

**vibe 인사이트**: vibe는 CLI 도구이므로 직접 적용 대상은 아니지만, **진행률 표시** 개선에 참고. Phase 2 이미지 다운로드 시 진행률을 더 효율적으로 표시할 수 있음.

### 10. 브리지 아키텍처

**Claude Code 패턴**: 로컬↔클라우드 연결, 최대 32개 병렬 세션, 메시지 중복 제거(순환 버퍼).

**vibe 인사이트**: 향후 vibe의 원격 실행/웹 대시보드 기능 구현 시 참고.

### 11. Bash 보안 (Tree-sitter 기반)

**Claude Code 패턴**: AST 분석으로 셸 명령어 구조 검증. 허용 목록만 통과시키는 "기본 거부" 설계.

**vibe 인사이트**: vibe의 `sentinel-guard.js`, `pre-tool-guard.js`가 유사 역할. 현재 패턴 매칭 기반인데, Tree-sitter 수준은 과도. 현재 방식이 적절.

### 12. 비용 추적 세분화

**Claude Code 패턴**: 모델별 입력/출력/캐시 토큰 분리 추적, 어드바이저 비용 재귀 합산.

**vibe 현황**: `logLlmCost`로 이미 잘 구현됨. 부족한 것은 **프로젝트별 예산 한도**와 **비용 경고**.
